6 Auth & roles
Admin admin
  • View all dashboards and reports
  • Upload CSV and manual reports
  • Invite new team members
  • Remove existing members
  • Change member roles
  • Enable / disable shareable link
  • Regenerate shareable link token
  • Save widget layout (persisted per user)
Viewer viewer
  • View all dashboards and reports
  • Change period / date range filters
  • Save their own widget layout
  • Cannot upload or edit data
  • Cannot access sharing.html
  • Cannot manage members or roles
  • Cannot manage share link
Share link access

A visitor arriving via /view/:token gets anonymous read-only access equivalent to Viewer — no login required. The server should render the dashboard with auth context role: "public", hiding all upload/admin controls. If is_active = false or the token doesn't exist, return a 404 or friendly "link no longer active" page.

Session / token suggestion

The frontend currently has no auth flow — all pages load without any session check. The recommended approach is cookie-based sessions (httpOnly, Secure) returned on login, with the server injecting the current user's role into the page as a JSON data attribute or small inline script. This avoids a full SPA refactor of the existing static HTML structure.

7 localStorage → server-side replacements
KeyUsed inCurrent value shapeServer replacement
subsocial-layout dashboard.html { rowOrder: string[], hidden: string[] } GET/PUT /api/layout — stored in widget_layouts per user. On page load, fetch and call applyState() with the response. On change, debounce the PUT 500ms.

No other localStorage keys are currently in use. All mock data (MOCK objects in platform pages) is hardcoded JS — there is no localStorage caching of API responses.

8 Upload flow detail
1
User selects platform + date range (Step 1)
Frontend holds selected platform_id + from/to dates in JS state. No API call yet.
2
CSV drop / file picker (Step 2, upload mode)
On file select, show filename + size. On "Preview →": POST /api/reports/upload with multipart form. Server validates headers against expected columns for the platform, rejects with 400 { error, missing_columns } if invalid, or returns 202 { report_id, status: "processing" }.
2b
Manual form (Step 2, manual mode)
All KPI fields for the selected platform are shown. On "Preview →": POST /api/reports/manual. Returns 201 immediately — no polling needed.
3
Review & confirm (Step 3)
Frontend polls GET /api/reports/:id until status = "ready" (or shows error). Displays summary of parsed metrics. On "Save report": update report status to confirmed, show success screen. Recommend polling interval: 1s, max 30s before timeout.
4
Success
Frontend shows success screen with links to the platform page and overview. No further API calls needed.
9 Sharing flow detail
Invite flow
1
Admin submits invite form
POST /api/team/invites with name, email, role. Server creates invite_tokens row, sends email with link https://app.com/invite/:token. Returns 201 { invite_id, email, role, expires_at }. Frontend appends to Pending Invites section.
2
Invitee clicks email link
Server validates token: not expired, not revoked, not already accepted. Shows an accept page (not yet built) with name pre-filled from invite.
3
Invitee sets password and accepts
POST /invite/:token/accept with { password }. Server creates user with pre-assigned role, marks token accepted_at, creates session, redirects to dashboard.
4
Admin revokes invite
DELETE /api/team/invites/:id sets revoked_at. Token immediately becomes invalid. Frontend removes the pending row.
Shareable link lifecycle
1
First load of sharing.html
GET /api/team/share-link returns current token and is_active. Frontend constructs display URL as https://subsocial.subvisual.co/view/{token}.
2
Toggle on/off
PATCH /api/team/share-link { "is_active": false }. Same token is preserved — toggling off just makes /view/:token return 403. Toggling back on re-activates the same URL.
3
Regenerate
POST /api/team/share-link/regenerate. Server sets invalidated_at on old row, inserts new row with new token. Old URL immediately 404s. Returns { token, url }. Frontend updates displayed URL.
10 Gotchas
⚠️
navigator.clipboard requires HTTPS
The copy button in sharing.html uses navigator.clipboard.writeText(). This API is unavailable on plain file:// or HTTP origins. It will silently fail. Ensure the app is served over HTTPS, or add a fallback using document.execCommand('copy') for local dev.
⚠️
Bounce rate delta is inverted
In website.html, a lower bounce rate is better. The delta badge is green when current < previous. The API must flag this: return a "invert_delta": true field for the bounce_rate KPI so the frontend can apply the correct colour logic. All other metrics treat increase as positive.
ℹ️
Chart updates use chart.update('active'), not destroy/recreate
All platform pages mutate chart.data.labels and chart.data.datasets[n].data directly, then call chart.update('active'). Do not destroy and recreate Chart.js instances on period change — the canvas will flash and animation state will be lost. Keep chart instances in module scope.
🔧
Members nav link is # on all platform pages
The "Members" nav item in the sidebar of linkedin.html, twitter.html, bluesky.html, youtube.html, dribbble.html, and website.html currently points to #. Update each to href="sharing.html".
⚠️
All chart data is Math.random() — re-renders on every page load
The wave(base, amp, len) helper in all pages uses Math.random(), so chart lines change on every load. Replace with deterministic data from the API. The function signature is a good guide for expected data shape: an array of len numbers centered around base with variation ±amp.
ℹ️
Custom date range falls back to 30D mock data
On all platform pages, selecting "Custom" shows date inputs but the setRange('custom') function falls back to MOCK[30]. The real implementation must add a from + to query param path: GET /api/platforms/:slug/metrics?from=2026-01-01&to=2026-02-15. The API should handle arbitrary ranges, not just 7/30/90.
ℹ️
Overview dashboard period tabs are not yet wired to data
dashboard.html has 7D/30D/90D/Custom tabs that only toggle an active CSS class — no data re-fetch on period change (unlike the platform pages which have full setRange() logic). The overview needs the same setRange() pattern added, calling GET /api/overview?period=N.
🔧
No auth guard on any page
All HTML files are currently static and open. Once auth is added, the server should return a 401 redirect to login for unauthenticated requests, and a 403 for Viewer attempts to access Admin-only endpoints. The "Preview as Viewer" toggle in sharing.html is a UI-only prototype — it does not test real auth logic.
